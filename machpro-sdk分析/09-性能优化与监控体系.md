# 第九章：性能优化与监控体系

> 一句话概括：从编译时、加载时、运行时三个阶段全链路优化，配合 Worker 线程分离计算密集任务和 metric-reporter 性能埋点实现可量化的性能治理。

## 9.1 编译时优化

### 9.1.1 JSX → 直接 DOM 操作

最核心的性能优化在编译阶段完成。传统 React 的渲染链路：

```
JSX → createElement → VDOM Tree → Diff → DOM Operations
```

MachPro 编译后的链路：

```
JSX → [编译时转换] → insert/attr/style/listen → DOM Operations
```

**省去的开销**：

1. `createElement` 调用的对象创建开销（每个元素一个 JS 对象）
2. VDOM 树的内存占用
3. Diff 算法的 CPU 消耗
4. Reconciliation 过程中的属性比较

### 9.1.2 Fragment 静态模板提取

静态 JSX 子树被提取为 Fragment，只在首次 `define` 时创建 DOM 节点。后续更新只执行 `render` 绑定动态值，不会重建 DOM。

性能收益：假设一个页面有 200 个 DOM 节点，其中 80% 是静态的，那么更新时只需处理 40 个动态绑定点，而不是 diff 200 个 VDOM 节点。

### 9.1.3 单字母别名

`insert/listen/attr/style/temp/child` → `i/l/a/s/t/c`，在高频调用场景下显著减少产物体积（一个中型页面约 500+ 次调用，每次省 5-6 字节，共节省 2.5-3KB）。

### 9.1.4 死代码消除

通过 `MACH_ENV` 环境变量替换 + Rollup tree-shaking，每端产物只包含该平台需要的代码：

```javascript
if (process.env.MACH_ENV === 'react') {
    // 这段代码在 H5/小程序构建时会被 tree-shake 掉
    initNativeRuntime();
}
```

## 9.2 加载时优化

### 9.2.1 代码注入顺序优化

rollup-plugin-injectjs 将基建代码 prepend 到业务代码前面，保证执行时序的同时也避免了额外的 HTTP 请求——所有代码打包在一个 IIFE 文件中。

### 9.2.2 异步子包加载

```typescript
// Mach.requireBundleAsync 带 Promise 缓存
Mach.requireBundleAsync(bundleId) {
    if (cache[bundleId]) return cache[bundleId];  // 缓存命中直接返回
    cache[bundleId] = new Promise((resolve) => {
        Mach.__requireBundleAsync__(bundleId, () => resolve(...));
    });
    return cache[bundleId];
}
```

首屏只加载主 bundle，次要模块通过 `requireBundleAsync` 按需加载，减少首屏体积。

### 9.2.3 QuickJS 字节码预编译

**包**：`mach_pro_tools_qjsc`

将 JS 代码预编译为 QuickJS 字节码（.qjsc 格式），客户端直接加载字节码跳过 JS 解析阶段。

性能收益：JS 解析通常占首屏时间的 10-30%，预编译可以大幅缩短冷启动时间。

## 9.3 运行时优化

### 9.3.1 insert 多层短路

```
insert(id, parent, value) 的优化链路：
  ① value === cache.originalValue → 直接返回（引用比较）
  ② array 相同引用 → 直接返回
  ③ 文本值相等 → 直接返回
  ④ 只有变化的元素才做 DOM 操作（delta diff）
```

### 9.3.2 attr 四层比较

```typescript
function attr(node, name, value) {
    const last = node.$attrData?.[name];
    if (last === value) return;                    // 第一层：引用比较
    if (shallowEqual(last, value)) return;         // 第二层：浅比较
    if (value == null) { removeAttribute; return; } // 第三层：空值处理
    node.setAttribute(name, value);                 // 第四层：实际操作
}
```

### 9.3.3 Style 缓存脏检查

```typescript
// style.ts
setProperty(name, value) {
    const strValue = String(value);
    if (this[name] !== strValue) {   // JS 侧缓存比较
        this[name] = strValue;
        this.style.setProperty(name, value);  // 变化时才跨 Bridge
    }
}
```

每次 style 更新先在 JS 侧做字符串比较，只有真正变化才跨 JS/Native 边界调用 `setProperty`。在高频更新场景（如动画）中，这个优化可以显著减少 Bridge 调用次数。

### 9.3.4 listen 事件替换优化

每个节点+事件名只注册一次 `addEventListener`，后续 handler 更新只替换 Map 中的值，避免 `removeEventListener` + `addEventListener` 的成本。

### 9.3.5 SCU 短路优化

```typescript
// processElementDeep 中
if (ctx.update(element) === false) {
    return target.$elementCache[processKey];  // SCU=false，直接返回缓存
}
```

## 9.4 Worker 线程支持

**包**：`mach-pro-worker`

在 Native 端 JS 引擎中实现 Web Worker 语义，将计算密集型任务（如列表数据预处理、JSON 解析）分离到 Worker 线程，避免阻塞 UI 渲染：

```javascript
const worker = new MachWorker('data-processor.js');
worker.postMessage({ type: 'processData', payload: largeData });
worker.onmessage = (result) => {
    store.dispatch('list/setData', result);
};
```

## 9.5 性能监控埋点

**包**：`mach-pro-metric-reporter`

自动采集和上报以下性能指标：

| 指标 | 采集点 | 说明 |
|------|--------|------|
| Bundle 加载时间 | JS 引擎 eval 前后 | JS 文件从下载到可执行的耗时 |
| 首屏渲染时间 | 首次 DOM 操作完成 | 从 bundle 执行到首个有效渲染 |
| 组件初始化耗时 | Component.$init 前后 | 单个组件的初始化时间 |
| 更新耗时 | $run 执行时间 | setState 触发的更新耗时 |
| Memory 占用 | 定时采样 | JS 堆内存使用量 |
| DOM 节点数 | 定时采样 | 当前页面 DOM 节点总数 |

## 本章小结

MachPro 的性能优化贯穿编译时（JSX→DOM 指令、Fragment 静态提取、死代码消除）、加载时（单文件 IIFE、异步子包、QJsc 字节码预编译）、运行时（insert 多层短路、attr 四层比较、style 缓存脏检查、listen 事件替换、SCU 短路）三个阶段。Worker 线程分离计算任务避免 UI 阻塞。metric-reporter 实现了性能指标的全链路自动采集和上报。

### 亮点话术

> "我们在三个阶段都做了深度优化。编译时通过 babel 插件将 JSX 转为直接 DOM 操作跳过 VDOM diff；加载时通过 QJsc 字节码预编译跳过 JS 解析阶段；运行时通过 insert 的 marker+cache 增量更新、style 的 JS 侧缓存脏检查减少跨 Bridge 调用。以 style 缓存为例，在动画场景下每帧可能触发 20+ 次样式更新，但真正变化的可能只有 2-3 个属性，JS 侧比较可以过滤掉 85%+ 的无效跨 Bridge 调用。"
