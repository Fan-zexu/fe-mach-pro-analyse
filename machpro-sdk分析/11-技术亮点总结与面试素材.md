# 第十一章：技术亮点总结与面试素材

> 将 MachPro 代码库的核心技术亮点提炼为 8 个面试级别的技术点，每个配有原理解析、源码佐证和面试话术。

## 11.1 八大技术亮点

### 亮点 1：编译时 JSX → 直接 DOM 操作（类 SolidJS）

**原理**：babel-plugin-mach-pro 在编译阶段将 JSX 转为 `insert/attr/style/listen` 等直接 DOM 操作指令，运行时跳过 VDOM 创建和 diff。

**源码佐证**：`mach-pro-core/src/dom/dom.ts` 导出的 insert/listen/attr/style/temp/child 六个函数及其单字母别名 i/l/a/s/t/c，就是编译器的输出目标。

**面试话术**：
> "与 React 的'运行时 VDOM diff'不同，我们在编译阶段就将 JSX 拆分为'静态模板'（Fragment.define，只创建一次）和'动态绑定点'（insert/attr/style，只更新变化值）。这种策略类似 SolidJS，但我们的 DOM 操作目标是 mach-pro-render 模拟的 Native DOM，最终通过 Element.super 调用驱动原生渲染引擎。"

---

### 亮点 2：Native DOM 模拟 + 双树镜像架构

**原理**：mach-pro-render 在客户端 JS 引擎中实现 Window/Document/Element 等 W3C DOM API 子集。Element 继承 Native Node 基类，JS 层维护 childNodes 数组用于 DOM 查询，Native 层维护渲染树用于实际渲染，通过 `super.appendChild/insertBefore/setAttribute` 同步两棵树。

**源码佐证**：`mach-pro-render/src/node/element.ts`（275 行），Element 类中所有 DOM 操作方法都包含 JS 侧状态维护 + super 调用 Native 对应方法。

**面试话术**：
> "我们采用了'双树镜像'架构——JS 侧维护一棵 childNodes 数组形式的 DOM 树用于查询和遍历，Native 侧通过 Node 基类维护渲染树。当 React 代码调用 appendChild 时，我们同时更新两棵树。这种设计让 React 代码可以像在浏览器中一样操作 DOM，但底层实际是 Native 渲染引擎在工作。文本节点做了折叠优化，不进入 Native 树。"

---

### 亮点 3：62 行 nextTick 替代 React Fiber Scheduler

**原理**：用 `Promise.resolve().then()` 实现微任务批量更新，62 行代码替代了 React 的整个 Fiber Scheduler + Lane Model。

**源码佐证**：`mach-pro-core/src/react/nextTick.ts`（62 行），`callbacks` 数组 + `pending` 锁 + `flushCallbacks` 批量执行。

**面试话术**：
> "React 18 的 Scheduler 支持多优先级、时间切片、可中断/恢复，但也带来了大量代码和运行时开销。MachPro 面向动态化页面场景，不需要复杂的优先级调度——所有 setState 都同等重要、需要尽快处理。所以我们借鉴 Vue 的 nextTick 思路，用 Promise.resolve().then() 实现微任务级批量更新，62 行代码搞定。flushing 标志保证同一微任务周期内的多次 setState 只 flush 一次。"

---

### 亮点 4：Proxy 劫持实现 Native Module 透明适配

**原理**：通过 ES6 Proxy 劫持 `Mach.requireModule` 的返回值，将 Native Module 调用透明重定向到 MSI API，实现 HarmonyOS 适配。2 级开关（Bundle + Module）支持灰度和降级。

**源码佐证**：`mach-pro-bridge-adaptor/src/index.js`（49 行），双层 Proxy + 12 个适配模块。

**面试话术**：
> "HarmonyOS 适配的核心挑战是：业务代码中大量使用 Mach.requireModule 调用 Native Module，而 HarmonyOS 的 Native API 完全不同。我们的方案是用 Proxy 劫持 requireModule，在返回的模块对象上再包一层 Proxy 拦截方法调用，透明重定向到 MSI API。整个方案对业务代码零侵入，通过 ABTest 开关可以按 Bundle 和 Module 两级灰度。"

---

### 亮点 5：marker + cache 增量 DOM 更新

**原理**：insert 函数通过 `parent.$cache[id]` 为每个"插入槽位"维护状态，更新时先做值引用比较短路，再通过 delta 启发式 diff 最小化 DOM 操作。

**源码佐证**：`mach-pro-core/src/dom/dom.ts` L81-L218，`insert` 函数 + `updateArray` 函数。

**面试话术**：
> "与 React 的 tree-level reconciliation 不同，我们做的是 slot-level 增量更新。编译器知道每个动态部分的位置，所以每个 insert 调用对应一个固定的'插入槽位'，通过 id 索引到缓存的上次状态。更新时先做引用比较（O(1)），相同就直接返回；不同才进入 diff。这种设计在'大部分内容不变、少量数据更新'的典型场景下特别高效。"

---

### 亮点 6：代码注入 + 多层 prepend 构建架构

**原理**：rollup-plugin-injectjs 通过 Rollup 的 `intro` hook 将 render 层、Preact、标签映射等基建代码按严格顺序 prepend 到业务代码前面，形成单文件 IIFE 产物。

**源码佐证**：`mach-pro-react-build/src/plugins/rollup-plugin-injectjs.ts`（230 行）。

**面试话术**：
> "Native 端的最终产物是一个 IIFE 格式的单文件。我们通过 rollup-plugin-injectjs 将多层基建代码按严格顺序注入：先初始化 DOM 运行环境（render），再挂载 React API（Preact），然后做标签映射（native-element），最后才是业务代码。顺序不能变——业务代码依赖 DOM 环境，DOM 环境依赖 Mach 全局对象。"

---

### 亮点 7：Preact Fork 的精准手术

**原理**：对 Preact 进行 10 处精准改动，覆盖事件系统（闭包替代 this）、样式适配（gradient 映射）、生命周期时序（wx.nextTick）、属性序列化（H5 Object）等。每处改动都有明确的平台适配原因。

**源码佐证**：`packages/preact/` 中散落在 diff/props.js、diff/style.js、compat/src/index.js 等文件的改动点。

**面试话术**：
> "Fork Preact 而非从零自研 H5 引擎的原因是：H5/小程序端需要更好的 React 生态兼容性，Preact 已经实现了 95% 的 React API。我们只做了 10 处精准改动，每处都是因为原版 Preact 无法通过配置或 options hook 解决的问题。比如事件监听系统改为闭包引用（因为小程序/Native DOM 模拟层的 this 指向问题），gradient 属性做三平台映射（Native 用 backgroundColor 接收渐变值），小程序的 commit 回调用 wx.nextTick 延迟（因为 setData 是异步的）。"

---

### 亮点 8：全局单例 Store + 跨 Bundle 共享

**原理**：通过 `globalThis.__MACH_PRO_STORE__` 确保主 bundle 和动态加载的子 bundle 共享同一个 store 实例。dispatch 通过 `"modelName/reducerName"` 字符串路由分发。

**源码佐证**：`mach-pro-store/src/rematch.ts`，全局单例模式 + dispatch 路由 + connect 浅比较 HOC。

## 11.3 深度追问预判

### "你认为 MachPro 的设计有什么不足？"

> "主要有几个方面值得优化。首先，双引擎虽然各自适配了最佳场景，但也意味着两套渲染 bug 需要修复，维护成本高。其次，Hooks 的 counter 索引方案虽然简洁，但在复杂组件中 counter 值不够稳定（比如条件分支中使用 hooks）。另外，列表 diff 算法的 delta 启发式在大列表随机乱序场景下可能不如 LIS 算法高效。最后，mach-pro-render 的文本节点折叠设计导致无法支持 `<text>` 嵌套 `<text>` 的富文本场景。"

### "如果要新增一个平台（比如鸿蒙 Next），需要做什么？"

> "主要三步：1) 在 bridge-adaptor 中新增适配模块，将 Mach Native Module 映射到鸿蒙 Next 的 API；2) 确保 mach-pro-render 的 Node 基类在鸿蒙 Next 的 JS 引擎中正确注入；3) 如果鸿蒙 Next 的渲染模型与 iOS/Android 有差异，可能需要在 native-element 中新增标签映射规则。构建系统基本不需要改动——MACH_ENV=react 的构建管线可以直接复用。"

## 11.4 技术对比总结表

| 维度 | MachPro | React | Vue 3 | SolidJS | React Native |
|------|---------|-------|-------|---------|-------------|
| VDOM | 无（编译时消除） | 有 | 有 | 无 | 有 |
| 更新策略 | insert 增量更新 | Fiber Reconciliation | Proxy 响应式 | 细粒度响应式 | Fiber + Bridge |
| 调度器 | nextTick (62行) | Scheduler (2000+行) | nextTick | 无调度器 | Scheduler |
| Hooks 存储 | counter 索引 | Fiber 链表 | setup 闭包 | 无 Hooks | Fiber 链表 |

| 状态管理 | 自研 ~200行 | Redux/Zustand 等 | Pinia | Stores | Redux/Zustand 等 |

## 11.5 一段话总结 MachPro

MachPro 是美团外卖团队自研的全页面动态化跨端渲染框架。它的核心创新是将编译时优化（类 SolidJS）与 Native DOM 模拟相结合：编译器在构建阶段将 JSX 转为直接 DOM 操作指令，运行时跳过 VDOM diff；mach-pro-render 在客户端 JS 引擎中构建了精简的 W3C DOM 运行环境，Element 类通过继承 Native Node 基类实现"双树镜像"同步。双引擎架构（Native 端自研 React 运行时 + H5/小程序端 Preact fork）兼顾了性能和生态兼容性。62 行的 nextTick 调度器、marker+cache 增量 DOM 更新、Proxy 劫持的 HarmonyOS 适配方案、多层代码注入的构建系统，共同构成了一套完整的从开发到部署的全页面动态化解决方案。
