# 第七章：跨端适配体系

> 一句话概括：通过 Proxy 劫持 + 模块适配 + 标签映射三层机制，实现一套代码在 Native(iOS/Android/HarmonyOS)、小程序、H5 三端透明运行。

## 7.1 Bridge Adaptor —— Native Module 适配

**文件**：`packages/mach-pro-bridge-adaptor/src/index.js`（49 行）

### Proxy 劫持原理

```javascript
const OriginalRequireModule = Mach.requireModule.bind(Mach);

Mach.requireModule = new Proxy(OriginalRequireModule, {
    apply(target, thisArg, args) {
        const [moduleName] = args;
        const originalModule = target(...args);
        
        // 检查是否需要适配
        if (!shouldAdapt(moduleName)) return originalModule;
        
        // 返回适配后的模块（也是 Proxy）
        return new Proxy(originalModule, {
            get(target, methodName) {
                if (adaptors[moduleName]?.[methodName]) {
                    return adaptors[moduleName][methodName];
                }
                return target[methodName];
            }
        });
    }
});
```

### 2 级开关控制

```
Bundle 级开关: ABTest 控制整个 bridge-adaptor 是否启用
Module 级开关: 每个适配模块可单独开关
```

当 `Mach.env.isHarmony === true` 时，适配器将 Mach 的 Native Module 调用重定向到 MSI（Multi-Service Integration）API。

## 7.2 标签适配体系

### mach-pro-native-element

**文件**：`src/elements-shim/jsx.ts`（33 行）

核心机制：Monkeypatch `React.createElement`，在 JSX 编译产物创建元素时做标签名透明替换。

```javascript
const ElementMap = new Map([
    ['div',    'view'],
    ['span',   'text'],
    ['img',    'image'],
    ['p',      'text'],
    ['a',      'text'],
    ['button', 'view'],
    // ...
]);

const oldCreate = React.createElement;
React.createElement = function(type, props, ...children) {
    if (typeof type === 'string' && ElementMap.has(type)) {
        type = ElementMap.get(type);
    }
    return oldCreate(type, props, ...children);
}
```

### mach-pro-h5-element

H5 端使用 Web Components（Custom Elements）实现 Native 标签的 H5 降级。

### mach-pro-h5-native-module

## 7.4 HarmonyOS 适配策略

HarmonyOS 适配是 bridge-adaptor 的核心场景。通过 `Mach.env.isHarmony` 检测运行环境后，所有 Native Module 调用走 Proxy 适配路径：

```
业务代码 → Mach.requireModule('WMNetwork')
         → Proxy 拦截
         → WMNetwork adapter
         → MSI.network.request()
         → HarmonyOS Native 实现
```

这种设计的优势是**业务代码完全无感知**——同一份代码在 iOS/Android 上直接调用 Mach Native Module，在 HarmonyOS 上通过 Proxy 透明重定向到 MSI API。

## 本章小结

MachPro 的跨端适配通过三层机制实现：1) bridge-adaptor 的 Proxy 劫持实现 Native Module 透明适配（特别是 HarmonyOS → MSI 的重定向）；2) native-element 的 createElement monkeypatch 实现标签名透明替换；3) h5-element/h5-native-module/wechat-native-module 实现各平台的 API 降级。整体设计保证业务代码对平台差异完全无感知。

### 亮点话术

> "在跨端适配设计上，我们采用了'Proxy 劫持 + 2 级开关'的方案。Proxy 劫持 `Mach.requireModule`，在返回的模块对象上再加一层 Proxy 拦截方法调用，透明地将 Mach Native Module 重定向到 MSI API。2 级开关（Bundle 级 ABTest + Module 级独立开关）保证了任何适配模块都可以独立灰度和降级。整个方案对业务代码完全透明——开发者继续写 `Mach.requireModule('WMNetwork').request()`，适配层自动处理平台差异。"
